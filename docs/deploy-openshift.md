# Deploying OpenClaw on OpenShift

The Kubernetes deployer creates all resources via the Kubernetes API. On OpenShift, it automatically adds an OAuth proxy sidecar so your instance is protected by OpenShift SSO — no cluster-admin required.

This guide covers OpenShift specifically. The deployer also works on plain Kubernetes (without the OAuth proxy, Route, and ServiceAccount).

For the full story — architecture, security model, and how the OAuth proxy works — see the [blog post](blog-installing-openclaw-on-openshift.md).

The installer uses the same `~/.openclaw` home layout as native OpenClaw:

- `~/.openclaw/workspace-*` for agent workspaces
- `~/.openclaw/skills` for shared skills
- `~/.openclaw/installer` for installer state

## Prerequisites

- An OpenShift cluster where you can create a namespace (no cluster-admin required)
- `oc` CLI authenticated (`oc login`) on the machine running the installer
- An API key or GCP service account for at least one model provider

**Storage:** OpenClaw uses SQLite, which requires POSIX file locking. Block storage classes (gp3-csi, managed-csi, thin-csi) work. Avoid NFS.

## Deploy

Start the installer on a machine where `oc login` (or `kubectl`) is authenticated to your cluster.

**No git clone required (Linux):**

```bash
curl -fsSLo run.sh https://raw.githubusercontent.com/sallyom/claw-installer/main/run.sh
chmod +x run.sh
./run.sh
```

**From source:**

```bash
git clone https://github.com/sallyom/claw-installer.git
cd claw-installer
npm install && npm run build && npm run dev
```

Open `http://localhost:3000`, pick **Kubernetes**, fill in the form, and hit Deploy. The installer detects OpenShift automatically.

### Deploy form

| Field | Example | Notes |
|-------|---------|-------|
| **Agent name** | `myagent` | ID for your default agent |
| **Owner prefix** | *(optional)* | Defaults to OS username. Combined with agent name: `alice-myagent-openclaw` |
| **Display name** | `My Agent` | Shown in the UI |
| **Image** | `quay.io/sallyom/openclaw:latest` | Container image |
| **API key** | *(your key)* | For Anthropic, OpenAI, or other providers |
| **Google Cloud Credentials (JSON)** | *(file upload or path)* | For Vertex AI — project ID auto-extracted |

## What Gets Created

The installer creates these resources in a dedicated namespace. See [`docs/examples/`](examples/) for annotated YAMLs.

| Resource | Name | Purpose |
|----------|------|---------|
| Namespace | `alice-myagent-openclaw` | Labeled for installer discovery |
| ServiceAccount | `openclaw-oauth-proxy` | OAuth proxy SA with redirect annotation |
| Secret | `openclaw-oauth-config` | SA token (client-secret) + cookie secret |
| Service | `openclaw` | Gateway (18789) + oauth-ui (8443) ports |
| Route | `openclaw` | TLS-terminated, targets oauth-proxy |
| PVC | `openclaw-home-pvc` | 10Gi for all persistent state |
| ConfigMap | `openclaw-config` | `openclaw.json` with gateway settings |
| ConfigMap | `openclaw-agent` | Agent workspace files |
| Secret | `openclaw-secrets` | Gateway token + provider API keys |
| Secret | `gcp-sa` | GCP SA JSON (Vertex AI only) |
| Secret | `openclaw-proxy-tls` | Auto-generated by OpenShift serving-cert |
| Deployment | `openclaw` | Init container + oauth-proxy sidecar + gateway |

On plain Kubernetes, the ServiceAccount, oauth-config Secret, proxy-tls Secret, and Route are omitted. The gateway binds to `lan` instead of `loopback`.

### Pod architecture

The Deployment runs three containers:

**init-config** — UBI9-minimal init container. Copies `openclaw.json` from the ConfigMap to the PVC, creates workspace directories, copies agent files, and sets GID 0 permissions for OpenShift's random UID assignment under `restricted-v2` SCC.

**oauth-proxy** (OpenShift only) — [OpenShift OAuth Proxy](https://github.com/openshift/oauth-proxy) sidecar on port 8443. Authenticates against OpenShift OAuth using SA-based OAuth (`user:info` scope, no cluster-admin). Forwards authenticated requests to the gateway on localhost:18789.

**gateway** — OpenClaw gateway. Binds to loopback on OpenShift (oauth-proxy fronts it) or `lan` on plain Kubernetes. Reads config from PVC, connects to model providers over HTTPS.

All containers run under OpenShift's `restricted-v2` SCC with no custom SCC required: non-root, read-only root filesystem, all capabilities dropped.

### How OAuth works without cluster-admin

OpenShift supports SA-based OAuth — a ServiceAccount acts as an OAuth client without a cluster-scoped `OAuthClient` resource:

1. ServiceAccount gets an `oauth-redirectreference` annotation pointing to the Route
2. Client ID is `system:serviceaccount:<namespace>:<sa-name>`
3. Client secret is a ServiceAccount token from the TokenRequest API
4. Scope is `user:info` — any authenticated OpenShift user can access

To restrict access to namespace members, switch to `--openshift-sar` (requires a `ClusterRoleBinding` for `system:auth-delegator`, which does need cluster-admin). See comments in [`src/server/deployers/openshift/oauth-proxy-container.yaml`](../src/server/deployers/openshift/oauth-proxy-container.yaml).

## Access Your Instance

The Route URL is printed in the deploy log:

```
https://openclaw-alice-myagent-openclaw.apps.your-cluster.example.com
```

OpenShift OAuth handles authentication — you'll be redirected to the login page. After authenticating, the Control UI asks for your **Gateway Token**, which was printed in the deploy log and saved to:

```
~/.openclaw/installer/k8s/alice-myagent-openclaw/gateway-token
```

## Updating Your Agent

After the initial deploy, your agent files live in `~/.openclaw/workspace-<prefix>_<name>/` on the host. To push changes to a running K8s deployment:

1. Edit the files locally (e.g., change `AGENTS.md` to update instructions, `SOUL.md` to change personality)
2. Go to the **Instances** tab and click **Re-deploy**

Re-deploy reads your local files, updates the `openclaw-agent` ConfigMap, and restarts the pod. The init container copies the updated ConfigMap contents into the PVC on startup.

**Stop/Start vs Re-deploy:** Stop and Start only scale replicas (0 and 1). They do *not* sync agent files from the host — the pod restarts with whatever is already in the ConfigMap and PVC. Use **Re-deploy** when you've changed agent files locally.

## Full Re-deploy

To change deploy configuration (API keys, model provider, image, etc.), fill in the deploy form and deploy to the same namespace. The installer uses create-or-replace logic on every resource, and the Deployment's `openclaw.io/restart-at` annotation forces a pod rollout.

The deploy config (with secrets redacted) is saved to `~/.openclaw/installer/k8s/<namespace>/deploy-config.json` after each deploy.

## Instance Management

The **Instances** tab finds all namespaces labeled `app.kubernetes.io/managed-by=openclaw-installer`. For each instance:

- Pod status, container state, restart count
- Route URL (OpenShift)
- **Re-deploy** — updates agent files from host to ConfigMap and restarts pod
- **Stop** — scales deployment to 0 (PVC preserved)
- **Start** — scales back to 1
- **Delete** — tears down all resources and deletes the namespace

## Teardown

From the Instances tab, click Delete. The installer explicitly deletes each resource before removing the namespace to avoid stuck Terminating states:

1. Deployment, Service, Route
2. Secrets (openclaw-secrets, gcp-sa, openclaw-oauth-config, openclaw-proxy-tls)
3. ServiceAccount (openclaw-oauth-proxy)
4. ConfigMaps (openclaw-config, openclaw-agent)
5. PVC (openclaw-home-pvc)
6. Namespace

## Host Filesystem

```
~/.openclaw/
├── installer/
│   └── k8s/                                 # K8s instance configs
│       └── <namespace>/
│           ├── deploy-config.json           # Deploy config (secrets redacted)
│           └── gateway-token                # Gateway auth token
├── skills/
│   └── <skill-name>/
│       └── SKILL.md
└── workspace-<prefix>_<name>/
    ├── AGENTS.md
    ├── SOUL.md
    └── ...
```

Edit files in `workspace-<id>/` and click **Re-deploy** from the Instances tab to push changes. The installer updates the ConfigMap and restarts the pod.

## Vertex AI

For Google Vertex AI or Anthropic Claude via Vertex AI:

1. Upload your GCP service account JSON in the deploy form (or provide an absolute path)
2. The installer extracts `project_id` automatically
3. Creates a `gcp-sa` Kubernetes Secret with the JSON
4. Mounts it at `/home/node/gcp/sa.json` in the gateway container
5. Sets `GOOGLE_APPLICATION_CREDENTIALS`, `GOOGLE_CLOUD_PROJECT`, and `VERTEX_ENABLED` env vars
